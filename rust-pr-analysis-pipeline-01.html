Pipeline Architecture & Process

How the PR analysis pipeline turns merged-PR review history into skill files · window 2025-09-18 → 2026-05-15

The pipeline mines merged Rust-touching PRs from flox/flox, classifies every substantive line-comment with a Haiku subagent, clusters the classifications into themed findings, and then synthesizes skill files / CLAUDE.md / gap reports from those findings. Every stage is idempotent, file-based, and survives subagent failures.

2. Stage-by-stage breakdown

2.1 Ingest

Pull raw PR + comment + code data from GitHub via the gh CLI (REST and GraphQL) and populate the SQLite tables.

Audit step: audit_coverage.py --ingest-only validates ID parity with GitHub before any downstream stage runs.

2.2 Classify

Convert each substantive line-comment into a structured record:

{ taxonomy, was_addressed, rule_statement, confidence, prompt_hash }

Two paths exist:

  • classify_via_subagent.py — subagent-orchestrated; no ANTHROPIC_API_KEY required because the parent Claude session dispatches Haiku subagents directly. This is the path used in production.
  • classify_comments.py — legacy direct Anthropic SDK path; requires ANTHROPIC_API_KEY. Kept for reference.

Subagent flow

  1. prepare mode reads pending non-noise line-comments and writes batches of 15 as /tmp/pilot_classify/batch_N.json.
  2. Controller dispatches parallel Haiku subagents; each subagent reads exactly one batch file and writes result_N.json.
  3. ingest mode reads the result files, validates each row, computes prompt_hash, and writes into the classification table.

2.3 Aggregate

Cluster classified comments into themed finding rows using semantic similarity.

Script: aggregate_findings.py

  • Embedding clustering: sentence-transformers all-MiniLM-L6-v2, cosine similarity at threshold 0.65, greedy single-pass.
  • AGENTS.md matching: key-token substring overlap — at least 3 distinctive tokens of length ≥ 4 appearing inside the same AGENTS.md section.
  • Finding scope: a finding is cross-cutting when cross_area_count ≥ 2 AND tier1_reviewer_count ≥ 1; otherwise it stays scoped to its dominant area.

2.4 Synthesize

Turn finding rows into human-readable skill files, CLAUDE.md content, and the gap report.

  • Model: Claude Sonnet 4.6, capped at ~4000 output tokens per call.
  • Each call is logged in synthesis_log with prompt_hash, raw response, and output path so synthesis is auditable and reproducible.
  • Citations are PR numbers drawn from finding.evidence_pr_numbers JSON arrays; the synthesizer is instructed never to invent PR numbers.

Outputs:

  • .claude/skills/flox-rust-review/SKILL.md
  • .claude/skills/flox-rust-stylistic-conventions/SKILL.md — NEW, extracted from the noise deep-dive
  • Three area-specific CLAUDE.md files (commands, models/environment, providers)
  • scripts/pr-analysis/findings/gap-report.md

2.5 Visualize

Regenerate human-readable HTML reports from the current DB state. All outputs are self-contained — inline SVG, no JavaScript, no CDN.

  • build_dashboard.py → rust-pr-analysis-dashboard-01.html
  • build_noise_deep_dive.py → rust-pr-analysis-noise-deep-dive-01.html
  • build_index_and_pipeline.py → this page + the index

4. Configuration knobs

The values locked in after three pilot iterations. Changing any of these invalidates the existing classification rows because prompt_hash changes.

Reviewer tiers and weights

  • T1 ysndr, mkenigs, dcarley — weight 3.0
  • T2 djsauble, gilmishal, billlevine — weight 2.0
  • T3 all other humans — weight 1.0
  • T4 bots (*[bot], Copilot) — weight 0.0, excluded

Hot areas

  • cli/flox/src/commands/
  • cli/flox-rust-sdk/src/models/environment/
  • cli/flox-rust-sdk/src/providers/

Noise filter patterns

  • URL-only bodies
  • Suggestion-block-only (just a ```suggestion diff)
  • lgtm / thanks / emoji-only acknowledgments
  • Praise/nit prefix with body ≤ 40 characters

Taxonomy (15 entries, seeded from AGENTS.md sections)

error-handling, provider-traits, type-safety, user-facing-messages, naming, testing, imports, manifest-usage, deprecated-patterns, logging-tracing, formatting-style, control-flow, semantic-correctness, ld-floxlib, other

Other

  • Cluster threshold: MiniLM cosine 0.65
  • AGENTS.md matching: key-token substring, ≥ 3 distinctive tokens, each ≥ 4 chars
  • Batch size: 15 comments per Haiku subagent (Iter 3 calibration)
  • Cross-cutting requirement: cross_area_count ≥ 2 AND tier1_reviewer_count ≥ 1
  • Confidence formula: 0.4 · tier_signal + 0.2 · min(evd/5, 1) + 0.2 · min(cross_area/3, 1) + 0.2 · acceptance_rate

5. Subagent orchestration

Why the subagent path exists: the analysis already runs inside a Claude Code session, so requiring users to also manage an ANTHROPIC_API_KEY would be an unnecessary integration burden. The subagent path piggybacks on the existing Claude entitlement.

How it works: the parent dispatches background Agent(model: "haiku") calls per batch file. Each subagent reads exactly one batch_N.json from /tmp/pilot_classify and writes exactly one result_N.json. Failed subagents leave their batch file untouched and can be retried.

Reproducibility: prompt_hash = SHA256(SYSTEM_PROMPT + "\n---\n" + taxonomy_block) is stored per classification row. Classifications from different prompt or taxonomy versions are distinguishable in the DB so synthesis can choose a coherent subset.

Coordination is file-based, not in-memory: subagent failures don't lose work; the controller can rescan /tmp/pilot_classify on every run.

6. Invariants enforced by audit_coverage.py

  1. ID parity — the set of comment IDs in the DB matches what GitHub reports per PR. Catches a partial ingest or a deleted comment.
  2. Snippet coverage — every line_comment has a comment_final_code row (the snippet itself may be NULL if the file was deleted, but the row must exist).
  3. Classification coverage — every non-noise, non-bot line_comment has at least one classification row. Skipped under --ingest-only.
  4. Area mapping — no cli/* path falls into area='other'. A failure indicates a missing prefix in lib/areas.py.

7. Known pitfalls and lessons

  1. INSERT OR REPLACE cascades. SQLite implements REPLACE as DELETE + INSERT, which triggers ON DELETE CASCADE on children. ingest_comments originally lost every classification on each re-ingest. Fix: INSERT … ON CONFLICT(id) DO UPDATE SET ….
  2. Subagent self-doubt. Haiku occasionally pre-emptively claims a Read might exceed token limits and refuses. Dropping the batch size from 30 to 15 eliminated the trigger entirely.
  3. Heuristic calibration swings. AGENTS.md matching went 100% → 0% → 73% across three iterations of “the same idea” expressed differently (substring → whole-word → key-token ≥ 3 of len ≥ 4). The middle iteration was as broken as the first; only the third was defensible.
  4. Window sampling matters. A 15-PR recent-window pilot showed 52% activation rate on a key heuristic — entirely because of one fix-series concentrated in that window. Always validate calibration on a different historical window before scaling to the full corpus.
  5. Stylistic rules are under-codified. 33% of stylistic-taxonomy rules are not in AGENTS.md, vs 27% overall. Reviewers enforce them but they live as tribal knowledge — which is why a dedicated flox-rust-stylistic-conventions skill is the right next deliverable.
