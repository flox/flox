Overview

This project mined 8 months of merged pull requests from the flox/flox repository to extract code-review-validated conventions for Rust development. The pipeline ran from raw GitHub API data through SQLite storage, noise filtering, LLM classification (Haiku subagents), Sonnet re-classification, embedding-based clustering, and multi-stage synthesis, ending with four new CLAUDE.md files and a gap report against the existing AGENTS.md.

Pipeline Architecture

The pipeline is a 10-script SQLite warehouse with an LLM classification layer and a separate synthesis pass. Each stage is idempotent and can be re-run independently.

scripts/pr-analysis/
├── schema.sql              10 tables: pr, line_comment, review_summary,
│                           pr_comment, comment_final_code, classification,
│                           finding, reviewer, pr_file, synthesis_log
├── init_db.py              --reset wipes DB; WAL/shm cleanup
├── ingest_prs.py           --since/--until/--limit/--rust-only/--all
├── ingest_comments.py      line-comments, noise filter, UPSERT semantics
├── ingest_review_summaries.py  pulls/:n/reviews body text
├── ingest_pr_comments.py   issue comments (top-level discussion)
├── ingest_final_code.py    ~40-line snippet at merge_commit_sha (cached)
├── ingest_thread_resolution.py  GraphQL reviewThreads
├── audit_coverage.py       4 hard invariants + informational counts
├── classify_via_subagent.py  prepare/ingest; dispatches Haiku subagents
├── build_sonnet_reclassify_input.py  selects evd≥2 comments for Sonnet pass
├── aggregate_findings.py   MiniLM cosine @ 0.65, key-token AGENTS.md match
└── lib/
    ├── db.py, reviewers.py, areas.py, taxonomy.py
    ├── classify_helpers.py  SYSTEM_PROMPT, prompt_hash (SHA-256)
    ├── noise_filter.py      URL-only / suggestion / lgtm / praise regexes
    └── gh.py               run_json, paginate_jsonl wrappers

Data flow

Ingest (gh CLI + GraphQL) → Noise filter (18% dropped) → Haiku classification (batches of 15, parallel subagents) → Sonnet re-classification (evd≥2 comments, quality pass) → MiniLM embedding clustering (cosine 0.65 threshold) → AGENTS.md key-token matching → Synthesis (area CLAUDE.md + skills + gap report)

Key engineering decisions

DecisionRationale
SQLite as the warehouseQueryable, portable, no server. schema.sql is the single source of truth.
Subagent-orchestrated Haiku classificationNo Anthropic API key required in the worktree. Subagents read batch files from disk.
INSERT … ON CONFLICT DO UPDATE instead of INSERT OR REPLACESQLite REPLACE = DELETE + INSERT, cascades to children. UPSERT preserves foreign-key children.
MiniLM embeddings for clusteringOne-line rule statements Jaccard at 0.3–0.4 even when paraphrasing the same rule; MiniLM cosine puts them at 0.55–0.65.
Key-token AGENTS.md matchingJaccard on full text swung 100% (trivially true) → 0% (trivially false). Key-token substring on per-section text landed at 73%.

Why Iter 4 showed 51% other

The recent window (Iter 3) was dominated by a single PR series (activation hint enhancements). That burst made 52% of comments fall in the activations area. The older window (Iter 4) had a broader mix of 36 PRs but fewer Tier-1 comments that produced extractable rules — many older comments were design questions or inline approvals that the noise filter did not drop. This recency-bias insight shaped the full-corpus strategy: run over the full 8-month window to dilute any single-month spike.

Drift Correction — Haiku Noise and the Sonnet Re-classification Pass

The rule_statement drift problem

Haiku subagents, when confronted with a comment they could not confidently categorize, fell back to a generic rule_statement such as:

"Review comment addressing code change."

This is not a rule — it is the model expressing uncertainty in rule-statement form. The other-cluster-candidates.txt output shows a cluster of 7 comments that all received this identical text, from four different areas (models/environment, activations, cli/other), all at confidence=0.50. That confidence floor is precisely the Haiku "I don't know" signal.

A second drift pattern was over-narrow rules. Haiku sometimes generated rules that described what the comment discussed rather than the convention the discussion implies. For example:

"question nonblocking: did you mean to remove all the comments? Seems like some could be stale but some might not be"

…became the rule statement verbatim (confidence 0.65), instead of the actionable convention "Preserve doc comments during refactors unless they are demonstrably stale".

The Sonnet re-classification pass

After the full-corpus Haiku run, a targeted Sonnet re-classification pass (build_sonnet_reclassify_input.py + a Sonnet subagent) reprocessed the 104 comments backing findings with evd≥2. These are the comments most likely to surface real conventions, so quality there matters most. Sonnet produced significantly higher confidence on naming and type-safety categories (avg 0.82 and 0.82 vs Haiku's 0.74–0.75) and correctly promoted several "other" classifications to specific taxonomies.

Reviewer Voices

Three Tier-1 reviewers together authored 698 of the 1,047 ingested review comments. Each has a distinct reviewing philosophy.

ysndr — Tier 1 — 225 comments ingested

ysndr's comments cluster around architecture and lifecycle correctness and design documentation. When reviewing authentication code, ysndr pushed back on sentinel values and missing variants, preferring that code carry richer semantic signal even in degraded states (an expired token is more informative than ""; a Credential::NoToken variant may be less clear than Option<Credential>). In activation code, ysndr introduced the async select!-based signal handling pattern, reflecting a preference for explicit shutdown sequences that correctly release resources. In provider code, ysndr flagged removed doc comments as a loss — documentation is a first-class deliverable, not a by-product. The throughline: write code that carries its own context, whether through richer types, richer comments, or richer error states.

mkenigs — Tier 1 — 305 comments ingested

mkenigs's comments reflect concern for reviewer and future-author usability. Several comments ask whether a change is related to the PR's stated goal — a signal that mkenigs expects PRs to be narrowly scoped and that surprises in the diff require explanation. mkenigs raised testing coverage repeatedly, asking for tests when bug fixes landed without them. In user-facing output, mkenigs flagged counterintuitive terminology and unclear man page structure. In Cargo management, mkenigs noted workspace version drift from an uncoordinated inline version. The throughline: make every change easy for the next reviewer to evaluate — tight scope, documented intent, tests where warranted, and consistent infrastructure conventions.

dcarley — Tier 1 — 175 comments ingested

dcarley's comments reflect operational and resource-pressure awareness. When reviewing the filesystem watcher, dcarley flagged unnecessary spinning on every file event as a resource concern. When reviewing blocking waits, dcarley raised the question of timeouts for stuck processes. When reviewing breaking changes, dcarley asked whether the disruption could be framed as a user benefit. In code style, dcarley flagged double negatives in shell scripts, labelled blocks, and long lines that would fail the linter. The throughline: think about what happens in production at scale — resource pressure, broken flows, and user-visible disruption are first-order concerns, not afterthoughts.

Lessons Learned

Eight things this project taught us about LLM-driven analysis pipelines.

  1. Plan bugs against external APIs are normal. gh pr list --json doesn't expose mergeCommitOid or author.type — both were assumed in the original plan. Implementer subagents that adapt cleanly and flag deviations are more valuable than ones that blindly follow a spec.
  2. Heuristic calibration is a swing problem. AGENTS.md matching went 100% (useless-true) → 0% (useless-false) → 73% (defensible) across three iterations of the same idea expressed three different ways. No heuristic is "obviously right" before you see failure modes. Budget for at least two retros before a full run.
  3. Embeddings are the right tool for one-line rule deduplication. Even paraphrased duplicates only Jaccard at 0.3–0.4; MiniLM cosine puts them at 0.55–0.65. The threshold (0.65) is empirical and corpus-specific — validate it on a sample before committing.
  4. INSERT OR REPLACE with ON DELETE CASCADE is a footgun. SQLite implements REPLACE as DELETE+INSERT, which cascades to all child rows. This silently wiped comment_final_code rows during a re-ingest. Fix: INSERT … ON CONFLICT(id) DO UPDATE SET …. Test cascade behavior before any bulk re-ingest that touches a parent table.
  5. Subagent self-doubt is real and batch-size-sensitive. A Haiku subagent failed on a 70KB batch claiming "exceeds token limits" while three siblings handled 58–74KB files fine. Reducing batch size from 30 to 15 items eliminated the trigger entirely — smaller batches mean less per-item context, less Haiku hedging.
  6. Calibrating on one time window risks overfitting. The recent 2-month window was 52% activations due to a single fix series. Iter-4 on a different window showed 51% other, not 31%. This is not a pipeline bug — it is a corpus-composition signal. Use a full multi-month window for the final run.
  7. The "other" bucket is not all noise. The Task 8.5 re-cluster of high-confidence other comments found 19 actionable rules that the Haiku classifier had given up on. A targeted Sonnet pass on evd≥2 comments reduced the other rate from 39% to 6% for that subset. Two-tier classification (cheap Haiku for breadth, expensive Sonnet for precision on flagged comments) is the right architecture.
  8. Evidence count is a weak signal at corpus scale. 474 of 488 findings have exactly one supporting comment. This is not pipeline failure — it is the nature of the flox review corpus: reviewers rarely raise the same abstract principle twice in the same words. Single-evidence findings backed by a Tier-1 reviewer who accepted the change are still meaningful signal; they are just "a reviewer enforced this at least once," not "a team-wide invariant."
